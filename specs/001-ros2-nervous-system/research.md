## 1. Docusaurus Setup Best Practices

### Research Question
How should we configure Docusaurus v3.6.x for a docs-only documentation site optimized for Module 1?

### Decision: Use Classic Theme with Auto-Generated Sidebars

**Chosen Approach**:
- **Theme**: `@docusaurus/preset-classic` (default theme)
- **Sidebars**: Auto-generated from file structure using `autogenerated` mode
- **Diagrams**: Static SVG/PNG files (no Mermaid plugin for v1)

**Rationale**:
1. **Classic Theme**:
   - Meets all constitutional requirements out-of-the-box (system fonts, WCAG AA, mobile-first, <300KB bundle)
   - Zero configuration needed for accessibility, responsive design, dark mode
   - Well-documented, stable, widely used (reduces maintenance burden)
   - Free and open-source (MIT license)

2. **Auto-Generated Sidebars**:
   - Simpler than manual `sidebars.js` configuration for Module 1 (only 3 chapters)
   - File-based navigation: sidebar reflects actual file structure
   - Easy to extend for future modules (add files, sidebar updates automatically)
   - Configuration example:
     ```js
     // sidebars.js
     module.exports = {
       tutorialSidebar: [
         {
           type: 'autogenerated',
           dirName: '.', // Auto-generate from /docs/ directory
         },
       ],
     };
     ```

3. **Static SVG/PNG for Diagrams**:
   - Full control over diagram appearance (no syntax limitations)
   - Offline rendering (diagrams work without JavaScript, faster page load)
   - Smaller bundle size (no Mermaid library ~50KB)
   - Better accessibility (can add detailed alt text and ARIA labels)
   - Tools: Excalidraw (free, simple, SVG export) or draw.io/diagrams.net (free, web-based)

**Alternatives Considered**:
- **Custom Theme**: Rejected (adds complexity, unnecessary for v1, violates simplicity principle)
- **Manual `sidebars.js`**: Rejected (more maintenance, manual updates when files change)
- **Mermaid Plugin**: Rejected for v1 (adds 50KB to bundle, syntax learning curve, less design control; can reconsider for interactive diagrams in future modules)

## 2. Markdown Authoring Conventions

### Research Question
Should we use pure Markdown or MDX (Markdown + JSX), and what frontmatter schema is required?

### Decision: Pure Markdown with Standardized Frontmatter

**Chosen Approach**:
- **Format**: Pure Markdown (`.md` files, not `.mdx`)
- **Frontmatter Schema** (as specified in spec FR-030):
  ```yaml
  ---
  title: "Chapter Title"
  description: "50-150 character summary of chapter content for SEO and previews"
  keywords: ["keyword1", "keyword2", "keyword3", "keyword4", "keyword5"]
  difficulty: beginner  # or intermediate, advanced
  estimated_time: "25 minutes"
  ---
  ```

**Rationale**:
1. **Pure Markdown**:
   - Simpler syntax (no JSX components to learn)
   - Faster parsing and build times (no JSX transformation)
   - Better compatibility with external Markdown tools (linters, editors, converters)
   - Sufficient for Module 1 content (no interactive elements needed)
   - MDX can be adopted in future modules if interactive components required (e.g., quizzes, calculators)

2. **Frontmatter Schema**:
   - Enables RAG metadata extraction (title, keywords for semantic search)
   - Supports content filtering (difficulty level for learner progression)
   - Provides user expectations (estimated reading time)
   - SEO optimization (description for search engines and social media previews)

**Docusaurus Markdown Features to Use**:
- **Admonitions** (callouts): For tips, warnings, notes
  ```markdown
  :::tip Key Concept
  ROS 2 is middleware, not an operating system.
  :::

  :::note Definition
  **Middleware**: Software layer enabling communication between distributed components.
  :::
  ```
- **Code Blocks** with syntax highlighting:
  ```markdown
  ```python
  import rclpy  # ROS 2 Python client library
  from rclpy.node import Node  # Base class for ROS 2 nodes
  ```
  ```
- **Internal Links**: Relative paths
  ```markdown
  See [ROS 2 Communication Primitives](./chapter2.md) for details on topics and services.
  ```

**Alternatives Considered**:
- **MDX**: Rejected for v1 (adds complexity, not needed for text-based chapters; can revisit for interactive modules)
- **Custom Frontmatter Fields**: Rejected (stick to spec-defined schema; avoid field sprawl)

## 3. Diagram Creation Workflow

### Research Question
What tools and formats should be used for creating simple, conceptual diagrams for Module 1 chapters?

### Decision: Excalidraw for SVG Diagrams with Style Guide

**Chosen Approach**:
- **Tool**: Excalidraw (https://excalidraw.com) - free, web-based, no account required
- **Format**: SVG (Scalable Vector Graphics)
- **File Size Limit**: < 100KB per diagram (optimized via SVGO or online compression)
- **Location**: `/docs/module1/assets/`

**Diagram Style Guide** (for consistency across chapters):
- **Colors**:
  - Primary: #3578E5 (Docusaurus blue) for key elements
  - Secondary: #25C2A0 (Docusaurus teal) for supporting elements
  - Neutral: #1C1E21 (dark gray) for text, borders
  - Background: #FFFFFF (white) or transparent
- **Fonts**: System fonts (Arial, Helvetica, or default Excalidraw font)
- **Line Style**: Solid lines for connections, dashed lines for optional/future elements
- **Iconography**: Simple geometric shapes (rectangles for components, circles for nodes, arrows for data flow)
- **Layout**: Left-to-right flow where possible (matches reading direction)

**Workflow**:
1. **Create Diagram** in Excalidraw:
   - Use simple shapes and arrows
   - Add text labels (keep concise, <5 words per label)
   - Follow style guide colors and layout
2. **Export as SVG**:
   - Excalidraw ‚Üí Export ‚Üí SVG
   - Check file size (should be <50KB unoptimized)
3. **Optimize SVG** (if >50KB):
   - Use SVGO (https://jakearchibald.github.io/svgomg/) to compress
   - Remove unnecessary metadata, whitespace
   - Target < 100KB final size (preferably <50KB)
4. **Add to Markdown**:
   ```markdown
   ![Diagram showing ROS 2 as the nervous system connecting AI (brain) to actuators (muscles)](./assets/nervous-system-analogy.svg)
   ```
5. **Write Alt Text**:
   - 1-2 sentences describing diagram for screen readers
   - Explain key relationships shown in diagram
   - Example: "Diagram illustrating the nervous system analogy for ROS 2: AI/perception layer (brain) communicates through ROS 2 middleware (nerves) to actuators and motors (muscles)."

**Diagrams for Module 1**:
- Chapter 1: `nervous-system-analogy.svg` (AI ‚Üî ROS 2 ‚Üî Actuators)
- Chapter 2 (multiple possible):
  - `ros2-node-system.svg` (nodes communicating via topics/services)
  - `topic-flow.svg` (publish/subscribe pattern with multiple subscribers)
  - `service-flow.svg` (request/response pattern, synchronous)
- Chapter 3 (multiple possible):
  - `humanoid-arm-links-joints.svg` (shoulder ‚Üí upper arm ‚Üí elbow ‚Üí forearm with link/joint labels)
  - `kinematic-chain.svg` (tree structure: torso ‚Üí arms, legs ‚Üí hands, feet)
  - `urdf-hierarchy.svg` (root link ‚Üí child links tree structure)

**Alternatives Considered**:
- **draw.io/diagrams.net**: Also free, more feature-rich, but heavier UI (Excalidraw preferred for simplicity)
- **Figma**: Powerful but requires account, not as accessible (rejected for free-tier accessibility)
- **Mermaid syntax**: Rejected (less design control, requires plugin, see Docusaurus Setup decision)
- **PNG format**: Rejected (larger file sizes, not scalable, SVG preferred)

## 4. Word Count Validation

### Research Question
How do we automate word count validation to enforce 2000-word chapter limit and 150-word paragraph limit?

### Decision: Automated Node.js Script in CI Pipeline

**Chosen Approach**:
- **Tool**: Custom Node.js script using `unified` + `remark` ecosystem
- **Execution**: Automated in GitHub Actions CI (fail build if limits exceeded)
- **Scope**: Count words in Markdown content, excluding frontmatter, code blocks, and image alt text

**Implementation**:

File: `scripts/validate-word-count.js`
```javascript
const fs = require('fs');
const matter = require('gray-matter'); // Parse frontmatter
const remarkParse = require('remark-parse');
const { unified } = require('unified');
const { visit } = require('unist-util-visit');

const CHAPTER_LIMIT = 2000;
const PARAGRAPH_LIMIT = 150;

function countWords(text) {
  return text.trim().split(/\s+/).filter(w => w.length > 0).length;
}

function validateChapter(filePath) {
  const content = fs.readFileSync(filePath, 'utf-8');
  const { content: markdown } = matter(content); // Remove frontmatter

  const ast = unified().use(remarkParse).parse(markdown);

  let totalWords = 0;
  const paragraphViolations = [];

  visit(ast, 'paragraph', (node, index, parent) => {
    const text = extractText(node);
    const words = countWords(text);
    totalWords += words;

    if (words > PARAGRAPH_LIMIT) {
      paragraphViolations.push({
        line: node.position.start.line,
        words,
        text: text.substring(0, 100) + '...'
      });
    }
  });

  // Also count words in list items, headings (exclude code blocks)
  visit(ast, ['listItem', 'heading'], (node) => {
    const text = extractText(node);
    totalWords += countWords(text);
  });

  return {
    filePath,
    totalWords,
    chapterLimitExceeded: totalWords > CHAPTER_LIMIT,
    paragraphViolations
  };
}

function extractText(node) {
  let text = '';
  visit(node, 'text', (textNode) => {
    text += textNode.value + ' ';
  });
  return text;
}

// Main execution
const chapterFiles = process.argv.slice(2); // Pass file paths as args
let hasErrors = false;

chapterFiles.forEach(file => {
  const result = validateChapter(file);
  console.log(`\nüìÑ ${result.filePath}`);
  console.log(`   Total words: ${result.totalWords}/${CHAPTER_LIMIT}`);

  if (result.chapterLimitExceeded) {
    console.error(`   ‚ùå ERROR: Chapter exceeds ${CHAPTER_LIMIT} words!`);
    hasErrors = true;
  } else {
    console.log(`   ‚úÖ Chapter word count OK`);
  }

  if (result.paragraphViolations.length > 0) {
    console.error(`   ‚ùå ERROR: ${result.paragraphViolations.length} paragraph(s) exceed ${PARAGRAPH_LIMIT} words:`);
    result.paragraphViolations.forEach(v => {
      console.error(`      Line ${v.line}: ${v.words} words - "${v.text}"`);
    });
    hasErrors = true;
  } else {
    console.log(`   ‚úÖ All paragraphs within limit`);
  }
});

if (hasErrors) {
  console.error('\n‚ùå Word count validation FAILED');
  process.exit(1);
} else {
  console.log('\n‚úÖ All word count validations PASSED');
  process.exit(0);
}
```

  },
  "scripts": {

```

**GitHub Actions Integration** (`.github/workflows/validate-content.yml`):

**Rationale**:
- **Automated Enforcement**: CI fails if limits exceeded (prevents merging non-compliant content)
- **Developer Feedback**: Run `npm run word-count` locally before committing
- **Excludes Non-Content**: Frontmatter, code blocks, image alt text not counted (focuses on actual prose)
- **Paragraph-Level Detection**: Identifies specific paragraphs violating 150-word limit (easier to fix)

**Alternatives Considered**:
- **Manual Validation**: Rejected (human error, inconsistent, not scalable)
- **Simple `wc` Command**: Rejected (doesn't exclude frontmatter/code blocks, no paragraph-level validation)
- **Online Tools**: Rejected (not automatable in CI, requires manual upload)

## 5. GitHub Pages Deployment

### Research Question
How should Docusaurus be deployed to GitHub Pages with automated CI/CD?

### Decision: GitHub Actions with Auto-Deploy on Merge to Main

**Chosen Approach**:
- **Workflow**: GitHub Actions (`.github/workflows/deploy.yml`)
- **Trigger**: Auto-deploy on push to `main` branch
- **Hosting**: GitHub Pages (`gh-pages` branch)
- **URL**: `https://<username>.github.io/<repo-name>/`

**Implementation**:

File: `.github/workflows/deploy.yml`
```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches:
      - main  # Deploy on merge to main
  workflow_dispatch: # Allow manual trigger

permissions:
  contents: write

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Fetch full history for git info

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build Docusaurus site
        run: npm run build
        env:
          NODE_ENV: production

      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./build  # Docusaurus build output directory
          cname: # Optional: add custom domain here if configured
```

**Repository Configuration**:
1. **Enable GitHub Pages**:
   - Go to repository Settings ‚Üí Pages
   - Source: Deploy from a branch
   - Branch: `gh-pages` / `root`
   - Save
2. **Update `docusaurus.config.js`**:
   ```js
   module.exports = {
     url: 'https://<username>.github.io',
     baseUrl: '/<repo-name>/',
     organizationName: '<username>',
     projectName: '<repo-name>',
     // ... rest of config
   };
   ```

**Deployment Flow**:
1. Developer merges PR to `main` branch
2. GitHub Actions triggers `deploy.yml` workflow
3. Workflow installs dependencies, builds Docusaurus site (`npm run build`)
4. Build artifacts in `/build/` directory deployed to `gh-pages` branch
5. GitHub Pages auto-publishes from `gh-pages` branch
6. Site available at `https://<username>.github.io/<repo-name>/` within 1-2 minutes

**Rationale**:
- **Free Tier**: GitHub Actions free tier (2000 minutes/month) sufficient (build <5 min per deployment)
- **Zero Configuration**: No external services, tokens, or paid accounts needed
- **Automatic Updates**: Content changes in `main` ‚Üí automatic deployment
- **Version Control**: `gh-pages` branch preserves deployment history
- **Fast Builds**: Docusaurus builds static HTML/CSS/JS in <2 minutes for Module 1 (3 chapters)

**Alternatives Considered**:
- **Manual Deployment**: Rejected (error-prone, not scalable)
- **Vercel/Netlify**: Rejected (requires external account, not pure GitHub workflow; can revisit if custom domain or preview deployments needed)
- **Custom Domain**: Deferred to future (GitHub subdomain sufficient for v1; can add CNAME later)

## 6. RAG-Friendly Content Structure

### Research Question
How should content be structured to optimize RAG (Retrieval-Augmented Generation) chatbot accuracy?

### Decision: Semantic Markdown with Glossary and Inline Definitions

**Chosen Approach**:
1. **Semantic Heading Hierarchy**: H1 (chapter title) ‚Üí H2 (main sections) ‚Üí H3 (subsections only if needed)
2. **Self-Contained Sections**: Each H2 section should be understandable independently (minimal forward/backward references)
3. **Inline Definitions**: Define technical terms on first use within each chapter (even if defined in earlier chapters)
4. **Glossary Page**: Separate `/docs/glossary.md` with all technical terms and definitions
5. **Frontmatter Metadata**: Keywords field used for semantic search hints
6. **Explicit Context**: Avoid pronouns without antecedents; repeat nouns for clarity (e.g., "ROS 2" instead of "it")

**Content Structure Best Practices**:

**Example: Self-Contained Section**
```markdown
## Topics: Publish/Subscribe Communication Pattern

**Topics** in ROS 2 are named channels that enable **publish/subscribe communication**. In this pattern:
- **Publishers** send messages to a topic without knowing who will receive them
- **Subscribers** listen to a topic and receive all messages published to it
- Multiple subscribers can listen to the same topic simultaneously (one-to-many communication)

**Use Cases for Topics**:
- Sensor data streams (cameras, lidars, IMUs) publishing continuously
- Status updates from robot subsystems (battery level, temperature)
- Asynchronous data flows where publishers and subscribers don't need synchronization

**Example**: A camera node publishes images to the `/camera/image` topic. Multiple subscribers (vision processing node, recording node, display node) can all receive the same images independently.

Topics differ from **Services** (covered below) in that they are asynchronous and do not expect a response.
```

**Why This Works for RAG**:
- **Chunk Boundary**: H2 heading creates natural semantic chunk for vector embedding
- **Self-Contained**: Section defines "topics," "publish/subscribe," "publishers," "subscribers" without assuming prior knowledge
- **Explicit Context**: Repeats "ROS 2" and "topics" rather than using "it" or "this pattern"
- **Cross-Reference**: Mentions "Services" with forward reference, but doesn't require reading it to understand topics
- **Keyword Density**: Repeats key terms ("topics," "publish/subscribe") for semantic relevance

**Glossary Structure** (`/docs/glossary.md`):
```markdown
---
title: Glossary
description: Definitions of technical terms used in Physical AI & Humanoid Robotics textbook
keywords: ["glossary", "definitions", "terminology", "ROS 2", "robotics"]
---

# Glossary

## A

### Action (ROS 2)
A communication pattern in ROS 2 for long-running tasks that provide feedback and can be canceled. Actions combine aspects of topics (feedback) and services (goal/result).

## M

### Middleware
Software layer that enables communication and coordination between distributed components in a system. ROS 2 is middleware, not an operating system.

## N

### Node (ROS 2)
An independent software process in ROS 2 that performs a specific task and can communicate with other nodes via topics, services, or actions.

## T

### Topic (ROS 2)
A named communication channel in ROS 2 using the publish/subscribe pattern for streaming data. Topics enable asynchronous, one-to-many communication.

[... continue alphabetically for all terms]
```

**RAG Chunking Strategy**:
- **Chunk Size**: Target 500-1000 words per chunk (aligns with H2 section size)
- **Chunk Overlap**: 100-150 words (capture context from previous/next section)
- **Metadata**: Include chapter title, section heading, keywords from frontmatter in chunk metadata
- **Semantic Embeddings**: Use OpenAI `text-embedding-3-small` or free alternative (Sentence Transformers)
- **Vector DB**: Pinecone free tier (100K vectors sufficient for entire textbook + future modules)

**RAG Retrieval & Response Prompt**:
```plaintext
You are an AI teaching assistant for the "Physical AI & Humanoid Robotics ‚Äî Essentials" textbook.

STRICT RULES:
1. Answer questions ONLY using the provided textbook content (retrieved chunks below).
2. Cite sources by chapter and section (e.g., "See Module 1, Chapter 2: ROS 2 Communication Primitives, Section: Topics").
3. If the answer is not in the provided content, respond: "This topic is not covered in the current edition of the textbook."
4. Do NOT hallucinate information or use external knowledge.
5. Match the textbook's tone: educational, precise, accessible (beginner-friendly).

RETRIEVED CONTENT:
[Insert top-k retrieved chunks with metadata]

USER QUESTION:
[User's question]

YOUR RESPONSE:
```

**Alternatives Considered**:
- **Glossary Only** (no inline definitions): Rejected (requires constant cross-referencing, breaks reading flow)
- **Inline Only** (no glossary page): Rejected (harder for RAG to retrieve isolated definitions, no quick reference for learners)
- **Longer Chunks** (whole chapters): Rejected (exceeds embedding model context limits, less precise retrieval)
- **Shorter Chunks** (paragraphs): Rejected (lose context, harder to generate coherent answers)
